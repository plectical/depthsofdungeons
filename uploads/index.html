Mobile Game Profiler

Real-time performance profiling for browser-based games — injectable into any page, no build step required.

Inject the Profiler

Option A — Script Tag (recommended for your own game)

Add this to the <head> or end of <body> in your game's HTML during development:

<script src="profiler.js"></script>

Copy profiler.js into your game's public/static folder. For slime-debug-react, drop it in /public/ and add the script tag to index.html.

Option B — Browser Console (any running page)

Open DevTools → Console, paste and run:

var s=document.createElement('script'); s.src='http://YOUR_LOCAL_IP:PORT/profiler.js'; document.head.appendChild(s);

Serve profiler.js locally: cd mobile-game-profiler && npx serve . — then replace the URL with your local IP shown in the serve output.

Option C — Bookmarklet (drag to toolbar)

Drag this link to your bookmarks bar. On mobile, save it as a bookmark then edit the URL to be the javascript: code.

► Inject Game Profiler

Drag to browser toolbar, then click it on any game page to inject the profiler.

Using the HUD

1
FPS Pill appears top-right

Shows current FPS (color-coded green/yellow/red) + mini sparkline of the last 60 seconds. Drag to reposition.

2
Tap the pill to expand the panel

Opens a full panel with 4 tabs: Metrics, Issues, Device, Export. The red badge on the pill shows error count.

3
Play your game normally

The profiler runs silently in the background, collecting data and auto-flagging issues as they occur.

4
Export a full JSON report

Go to the Export tab and download or copy the report. Share it for offline analysis, or paste into the browser console to inspect.

What It Detects

📉
FPS Drops

Flags when FPS falls below 45 (warn) or 25 (critical). Shows avg, min, sparkline history.

🧱
Main Thread Blocking

Long tasks >100ms that cause jank. Uses native Long Task API on Chrome, rAF delta fallback on iOS.

🔴
JS Crashes & Errors

Catches all uncaught exceptions and unhandled promise rejections with file + line info.

📈
Memory Leaks

Samples heap every 5s. Flags if heap grows >25% in 30s or exceeds 80% of the browser's heap limit.

🌡️
Thermal Throttling

Detects when the device overheats and throttles the CPU — FPS degrades >35% in a 10s window.

👆
Input Lag

Measures time from touchstart to next rAF frame. Flags if consistently >100ms.

📐
Layout Shifts (CLS)

Cumulative Layout Shift from the Web Vitals spec. Indicates unwanted reflows during gameplay.

🎮
WebGL Context Loss

Detects when the GPU context is forcibly revoked by the OS (common on mobile under memory pressure).

📱
Device Issues

Low RAM, high DPR, slow network, low battery, low-end GPU — all flagged at session start.

🌐
Network Conditions

Reports connection type (4G/3G/2G), downlink speed, RTT, and Data Saver mode status.

Issues Reference

ERROR
Critical FPS (<25)Game is nearly unplayable. Look for runaway loops, synchronous XHR, or heavy DOM work on the main thread.
ERROR
JS ErrorUncaught exception — may halt game logic. Fix or catch all exceptions. On cross-origin scripts you'll see "Script error." — enable CORS headers.
ERROR
WebGL Context LostOS reclaimed the GPU context — game will freeze. Add a webglcontextrestored handler to reload assets and resume rendering.
ERROR
Critical Memory (>80% heap)Browser tab is about to crash. Audit asset loading, dispose unused textures/buffers, and avoid holding large arrays in memory.
WARN
Low FPS (<45)Noticeable lag. Profile the main thread in Chrome DevTools Performance tab. Common causes: physics loop not using delta-time, unthrottled resize handlers, canvas overdraw.
WARN
Long Task (>100ms)Main thread was blocked — input events and rendering stall. Break heavy work into chunks using setTimeout(0), Web Workers, or scheduler.postTask.
WARN
Memory Leak (>25% growth / 30s)Heap is growing without being freed. Common causes: event listeners not removed, retained DOM nodes, unbounded arrays/maps, missing WebGL buffer disposal.
WARN
Thermal ThrottlingDevice is hot and slowing the CPU down. Reduce your rAF work budget, lower particle counts, or add a frame-rate cap to preserve thermal headroom.
WARN
Input Lag >100msTouch feels sluggish. Ensure touch handlers are passive: true where possible, move processing off the main thread, and avoid long tasks during touch response.
WARN
High DPR (≥3×)Canvas may be rendering at 3× the display pixels. Cap with Math.min(window.devicePixelRatio, 2) for significant GPU savings on high-DPI phones.
WARN
Low Device RAM (≤2GB)High risk of tab kill on Android & iOS. Minimize texture sizes, use compressed formats (WebP, ASTC/ETC2 for WebGL), and lazy-load assets.
WARN
Low-end GPU DetectedMali-400, Adreno 3xx, or PowerVR SGX chipsets have limited fill rate and shader capability. Reduce fragment shader complexity and draw calls.
INFO
Long Task API UnavailableiOS Safari doesn't support the Long Task Observer. rAF delta gaps are used as a fallback — less precise but still useful for catching severe blocking.

Common Mobile Performance Fixes

ProblemFix
FPS drops during combat/effectsProfile the rAF callback with Chrome DevTools → Performance → Record. Look for large yellow (scripting) blocks. Move particle/physics math to a Web Worker.
Memory grows each fightEnsure all event listeners are removed when components unmount. Check for retained canvas 2D/WebGL buffers, forgotten setInterval/setTimeout, and growing React state arrays.
Input lag on touchAdd touch-action: none to the canvas CSS. All touch listeners that don't call preventDefault should be { passive: true }. Move input processing out of the touch handler into the rAF loop.
iOS Safari tab crashMonitor heap via profiler export + memSamples. iOS allows ~1.2–1.5GB before killing the tab. Use compressed textures, reduce canvas resolution on low-RAM devices, and dispose WebGL resources explicitly.
High DPR overdrawcanvas.width = Math.floor(canvas.offsetWidth * Math.min(devicePixelRatio, 2)). Cuts render workload by ~44% on 3× devices.
Thermal throttle after ~5 minAdd a frame-rate cap: only call game logic every Nth rAF (e.g., cap at 30fps during idle). Reduce shadow/particle effects during prolonged sessions.
Slow first load on 3G/4GLazy-load non-critical assets. Use a Service Worker to cache everything after first load. Compress assets (WebP images, gzip/brotli JS).
CLS from UI panelsSet explicit width and height on all images/sprites. Avoid inserting DOM nodes above existing content. Use transform for animations instead of layout properties.

Console API

The profiler exposes window.__GP__ in the browser console:

__GP__.show()Open the HUD panel
__GP__.hide()Collapse to pill
__GP__.toggle()Toggle open/closed
__GP__.export()Download the JSON report
__GP__.report()Return the report object (inspect in console)
__GP__.stateLive state object — all raw metric data
// Inspect live state in console: console.table(__GP__.report().summary) // Watch FPS in real time: setInterval(() => console.log('FPS:', __GP__.state.currentFps), 1000) // Export after a combat session: __GP__.export()

Platform Notes

iOS
Safari (iOS)No Long Task Observer → rAF delta used as fallback. No performance.memory → heap metrics unavailable. No navigator.deviceMemory. WebGL context loss is more frequent than Chrome. Tab crash limit ~1.2–1.5GB.
Chrome
Chrome (Android)Full metric support: Long Task API, performance.memory, Layout Shift, navigator.deviceMemory, navigator.connection. Use this for the most complete profiling data.
Samsung
Samsung BrowserChromium-based — most Chrome APIs work. May have different JS engine tuning. Check if game uses any Samsung-specific APIs.
Firefox
Firefox (Android)No performance.memory or navigator.deviceMemory. Long Task API may be behind a flag. FPS and error tracking still work.
